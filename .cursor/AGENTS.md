# SpeedVQA — Master 指令中枢

本文件为仓库内 **Agent / 协作总入口**。开始任务时优先通读本节与「资源地图」，再按需打开规则与文档。

## 1. 环境与边界

- **仓库根**：以当前打开的 SpeedVQA 工作区为准（例如 `.../SpeedVQA`）。脚本、配置路径均相对于该根目录理解。
- **Git**：**仅在本仓库目录内** 执行 `git`；不要进入其他项目目录对其做提交、合并或推送。
- **验证**：具备终端与网络时，应 **自行执行** 命令完成检查与修复；避免只罗列「让用户去跑」的步骤（复杂交互或需本地凭证的除外）。

## 2. 资源地图

| 路径 | 用途 |
|------|------|
| **`.cursor/AGENTS.md`** | 本中枢（总览与优先级） |
| **`.cursor/rules/*.mdc`** | Cursor 规则：`alwaysApply` / `globs` 分层约束 |
| **`.cursor/skills/`** | 项目级技能说明或链接（可选，与全局 `~/.cursor/skills` 区分） |
| **`.cursor/workflows/`** | 可重复的步骤清单（发版、冒烟、数据集准备等） |
| **`plan.md`** | **大于 5 个 todo** 的复杂任务：先写计划、执行中同步、结束后收口 |
| **`CHANGELOG.md`** | 发布记录；**新条目在最上方**；tag 命名见该文件「Tag 规范」 |
| **`docs/`** | 设计与使用文档（与实现对齐的实操以 `docs/02_使用.md` 等为准） |
| **`archive/`** | 归档文档命名：`YYYYMMDD_HHMMSS_功能_模块说明.md` |

## 3. 发布与版本（标准化提交）

1. 更新 **`CHANGELOG.md`**（逆序，最新在上），条目注明计划使用的 **`tag N.MMDD.ABCD`**（段含义见 CHANGELOG 表格）。
2. **`git commit`** 主题与正文应与 CHANGELOG 对应条目一致。
3. 使用 **`git tag -a`** 打附注 tag，说明与 CHANGELOG 一致。
4. 需要远端同步时再 **`git push`** 分支与 tag。

## 4. 代码与任务原则（摘要）

- **范围**：只改当前任务需要的文件；避免无关重构与「顺手清理」扩大 diff。
- **风格**：修改前阅读周边代码；命名、类型、注释粒度与现有文件保持一致。
- **大任务**：任务拆成多项时，若预估 **超过 5 个 todo**，先在 **`plan.md`** 写清范围与验收，执行后更新 **`plan.md`** 与 **`CHANGELOG.md`**。

## 5. 对话与产出

- 结合上下文理解用户意图；最新一句常是对进行中任务的细化，而非全盘换题。
- 向用户说明时：**中文**为主；引用已有代码用 Cursor 要求的 ` ```start:end:path ` 格式。
- 任务结束简要总结：**已完成项** 与 **仍待办项**（若有）。

## 6. 扩展本中枢

- **Always-on 规则**：在 `.cursor/rules/` 新增 `*.mdc`，`alwaysApply: true`，保持 **简短、单主题**（建议每条 < 50 行）。
- **按文件触发**：同一目录下用 `globs` 限定范围，避免一条规则塞满全库。
- **新流程**：在 `.cursor/workflows/` 增加 Markdown 步骤说明，并在上表「资源地图」中 **补一行** 登记，避免孤立文档。
